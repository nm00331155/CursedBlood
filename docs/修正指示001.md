Godot 4.6.1 (.NET/C#) の CursedBlood プロジェクトに対して、モバイル縦画面向けの「視認性改善・操作改善・画面拡大」を一括実装してください。

# 最重要方針
現状の問題はキャラ絵そのものより、以下です。
1. 地形が読めず、何が掘れるか分からない
2. 進行可能方向や掘削可否が分からず、操作性が悪化している
3. プレイ画面が遠すぎて、キャラ・地形・掘削対象・重要情報が小さすぎる
4. HUD / ボタン / 操作説明などの文字が全体的に小さく、スマホで読みにくい
5. 下部説明文が常時表示され、プレイエリアを圧迫している

今回の修正では、まず「遊んで理解できること」と「スマホで見えること」を優先し、67x87 全域を常時見せることよりも、プレイヤー周辺の読解性と操作感を優先してください。

# 実装目的
- プレイヤーと地形を見やすくする
- 掘れる / 硬い / 掘れない を一目で分かるようにする
- 入力方向に対して「進めるか」「掘れるか」「詰まっているか」を事前に見せる
- カメラを寄せてプレイ感を改善する
- HUD・ボタン・テキストをスマホで読めるサイズに拡大する
- 動的バーチャルパッドの使い勝手を改善する
- 迷った時に必要以上にスワイプしなくても済むようにする
- ソナーや方向誘導の基礎を入れて、行き先判断を補助する

# 前提
- Godot 4.6.1
- C#
- 1080x1920 portrait
- モバイル前提
- プレイヤーは 8方向移動
- セルサイズ 16px
- byte array / chunk 管理ベース
- 本体とドリルは別表示にしている、または今後別表示にする想定
- 今は見た目の完全一致より、プレイ感と可読性を優先する

# 対象
既存の以下に関連するクラス / シーン / UI を必要に応じて修正してください。
- GameCamera.cs
- HUDManager.cs
- PlayerController.cs
- VirtualPad.cs（存在しないなら追加）
- ChunkManager.cs
- DigHelper.cs
- GameManager.cs
- RecoveryPoint / Sonar / Debug関連クラス（必要なら追加）
- CursedBloodMain.tscn またはメインゲームシーン
- 必要なら新規に Overlay / Highlight / DirectionHint 用の Node2D / Control を追加

# 実装内容

## 1. プレイ画面のズーム改善
### 目的
プレイエリアが遠すぎて、キャラ・地形・掘削対象が小さすぎる問題を改善する。

### 対応
- Camera2D のズームを見直して、現状より明確に寄せる
- まず 1.3倍前後寄った見た目になるよう調整する
- プレイヤー中心の追従を維持する
- 必要なら少し先読み（LookAhead）を入れて、移動方向前方が少し見えるようにする
- 上下のHUDと干渉しにくいよう、カメラ位置補正または表示中心補正を行う
- ズーム値は inspector / config から調整可能にする

### 受け入れ条件
- キャラが現状より明らかに見える
- セル種別が今より見分けやすい
- プレイエリアが狭くなっても「今必要な情報」が読みやすくなっている

---

## 2. HUD・フォント・ボタンの可読性改善
### 目的
スマホで文字が小さく読めない問題を改善する。

### 対応
- HUD の全フォントサイズを全体的に引き上げる
- 小ラベル、本文、数値、ボタン文字のサイズ階層を整理する
- 目安として現状比 1.2〜1.5倍程度を基準に調整する
- 数字情報（酸素 / 深度 / 借金 / HP など）は特に視認性を高める
- 下部の小さい操作説明テキストを常時表示しない
- 操作説明は以下のいずれかに変更する
  - 初回のみ表示
  - ヘルプ画面へ移動
  - 折りたたみ
  - Debug表示ON時のみ
- 結果画面、タイトル、スタート、報告書系の文字も小さければ拡大する
- ボタン・重要ラベルの padding / margin / panel サイズも調整して、窮屈に見えないようにする

### 受け入れ条件
- スマホ前提で、主要文字が無理なく読める
- プレイ中に不要な説明テキストが常時出ていない
- HUD が少し大きくなっても、全体として前より見やすい

---

## 3. 地形の視認性改善
### 目的
メインの掘削対象フィールドが分かりづらく、何が何だか分からない問題を改善する。

### 対応
セルタイプごとに明度・色相・コントラストを見直し、「瞬時に識別可能」にしてください。

最低限、以下を視覚的に区別できるようにする。
- Empty / 空洞
- Dirt / 掘りやすい
- Stone / 硬い
- HardRock / さらに硬い
- Bedrock / 掘れない
- Ore / 価値資源
- RecoveryPoint / 回収ポイント
- 危険物 / 敵 / 特殊対象（存在する場合）

### 表現方針
- 色差だけでなく、明度差・輪郭差・必要なら模様差も入れる
- 掘れる土は比較的明るい
- 硬いものは灰色寄り / 青灰寄り / 輪郭強め
- 掘れないものは明確に別物に見える濃色
- 空洞は背景色をはっきり区別
- 金や資源は遠目でも判別可能なアクセント色にする
- プレイヤー周辺数マスを少しだけ明るくする / vignette を弱くするなど、近傍優先の視認補助を入れてもよい

### 受け入れ条件
- プレイ中に「どれが掘れる土か」「どれが硬いか」「どれが進行不可か」が前より明確
- 遠景よりもプレイヤー近傍の判断がしやすい

---

## 4. 掘削可否ハイライト / 前方プレビューの追加
### 目的
「この方向に進めるのか」「掘れるのか」「壊せないのか」が分からない問題を改善する。

### 対応
プレイヤーの現在入力方向に応じて、実際に掘削判定される前方範囲を半透明でプレビュー表示してください。
掘削範囲は現在の DigHelper / player size / dig width に整合させること。

### 表示ルール
- 掘れる: 半透明の緑
- 硬いが掘れる / 速度低下あり: 半透明の黄
- 掘れない / Bedrock / 進行不可: 半透明の赤
- 必要なら前方輪郭線も表示
- 表示は現在方向に追随
- 停止中も最後の方向を維持してよい

### 実装要件
- DigHelper の実判定と一致したプレビューにする
- 表示専用 Overlay ノードを追加してもよい
- 描画負荷に注意し、プレイヤー周辺の必要範囲のみ描く
- Debug ではなく通常プレイ向け機能として実装する
- ON/OFF 切替可能ならなおよい（設定 or debug）

### 受け入れ条件
- プレイヤーは入力前/入力中に「この方向は意味があるか」を判断できる
- 無駄なスワイプが減る

---

## 5. 進路候補 / 方向ヒントの追加
### 目的
行き止まりや硬い地形で詰まった際、どこに進めば良いか分からない問題を改善する。

### 対応
プレイヤー周囲8方向について、最低限の進路ヒントを表示する。
以下のいずれか、または組み合わせで実装する。
- 進める方向に小さな矢印
- 掘れる方向に小さな方向マーカー
- 完全不可方向は非表示または赤点
- プレイヤー周囲のリングUIで 8方向状態表示
- ソナー反応時は特に有効方向を優先表示

### 推奨
- シンプルな 8方向インジケータをプレイヤー近傍に表示
- 進行可能: 白
- 掘削可能: 黄
- 不可: 非表示 or 赤
- 強い反応がある方向は少し点滅

### 受け入れ条件
- 詰まった時に、行ける方向の候補が前より分かる
- あっちこっちに無駄にスワイプしなくて済む

---

## 6. ソナーの基礎UI導入 / 強化
### 目的
回収ポイントや重要対象への誘導が弱く、判断材料が足りない問題を改善する。

### 対応
ソナー反応をプレイヤーに分かる形で画面上へ表示する。
最初は簡易版でよい。

### 反応段階例
- 遠い: 何かある表示のみ（方向不明）
- 中距離: だいたいの方向が分かる
- 近い: 明確な方向表示
- かなり近い: アイコン表示 or 強調表示

### UI案
- プレイヤー周囲に波紋
- 画面端矢印
- プレイヤー近傍の方向インジケータ
- 反応強度に応じた色 / 点滅 / 音

### 受け入れ条件
- 回収ポイントや重要物の方向が多少なりとも分かる
- 「今戻るか / 探すか」の判断がしやすくなる

---

## 7. 動的バーチャルパッドの改善
### 目的
動的パッドを入れても操作性が悪い問題を改善する。

### 問題仮説
- 方向が定まらない
- 進行候補が分からないので無駄にスワイプする
- パッド基点が安定せず、指位置がズレ続ける
- 少しのブレで方向が変わりすぎる

### 対応
- タッチ開始地点でバーチャルパッドを生成 / 表示
- ドラッグ中は基点を固定する
- 毎フレーム再センタリングしない
- 8方向スナップを明確にする
- デッドゾーンを設ける
- 小さな指ブレで方向が頻繁に変わらないようヒステリシスを入れる
- 必要なら「直前方向を少し維持」する
- 不可方向しかない場合でも、最寄り有効方向へ軽く吸着するオプションを実装してよい（やりすぎ注意）
- 透過率設定を残す / 実装する
- 固定パッドではなく、動的出現型を維持する

### 受け入れ条件
- 指の位置が流れにくい
- 方向が安定する
- 無駄な入力ブレが減る
- 少なくとも現状より操作しやすい

---

## 8. プレイヤー視認性の改善
### 目的
キャラクターが画面内でほぼ見えない問題を改善する。

### 対応
- カメラズームの改善に加え、プレイヤー自体にも視認補助を入れる
- 白または淡色のアウトライン
- 軽い発光
- 足元影
- 必要なら方向インジケータ
- 地形に埋もれないよう、プレイヤー描画優先度も見直す

### 注意
キャラアートの作り込みより、まずは位置認識しやすいことを優先する

### 受け入れ条件
- プレイヤー位置を一瞬で見失わない
- 地形と埋もれにくい

---

## 9. 常時表示情報の整理
### 目的
画面が狭いのに不要情報が常時表示され、読みづらさと圧迫感が出ている問題を改善する。

### 対応
プレイ中に常時出す情報を厳選する。
優先して残す候補:
- 酸素 / 残り時間
- 深度
- 借金 or 所持金（優先度高い方）
- HP
- 状態（Stable / Worn / Critical）
- ソナー反応 / 回収ポイント方向
- 取得情報（必要最小限）

削減候補:
- 詳細な操作説明
- debug系ショートカット
- 長文テキスト
- 常時不要な補足

### 受け入れ条件
- プレイ中に不要な文章が減る
- プレイエリアと主要情報が前より見やすい

---

# 実装上の注意
- 既存コードを壊しすぎず、段階導入可能な構造にする
- public / exported で調整可能にして、後で数値をいじりやすくする
- null参照を避ける
- 方向名や色定義は定数化 / enum化して管理する
- パフォーマンスに配慮し、毎フレーム重い処理をしない
- 描画は必要範囲に限定する
- 掘削プレビューは実際の DigHelper 判定と一致させる
- モバイル縦画面前提で、タップ領域と文字サイズに配慮する
